# 进球数预测模型四层十维重构计划

> **Goal:** 将进球数预测从当前的"统计+LLM双轨松散参考"升级为"四层十维深度融合决策引擎"，系统性解决联赛特征组不一致、球队级数据缺失、统计与基本面割裂三大核心问题。

**Analysis Date:** 2026-05-03
**Status:** 📋 待执行

---

## 一、当前进球数预测系统的致命缺陷

### 🔴 缺陷 1：联赛特征组（Cluster）定义散落三处且不一致

**影响范围**：`goals_predictor.py`、`temp_analyze.py`、`1_Dashboard.py`

| 文件 | 日职归属 | 韩职归属 |
|:---|:---|:---|
| `GoalsPredictor.get_statistical_prediction()` | D组（日韩孤立型） | D组 |
| `temp_analyze.assign_cluster()` | D组（日韩孤立型） | D组 |
| `1_Dashboard.py` 汇总 Tab | A组（开放型） | B组（防守型） |

**后果**：Dashboard 展示的特征组与统计匹配时使用的特征组不一致，用户在界面上看到的和后台算法用的是两套分组，导致统计信号指向混乱。

### 🔴 缺陷 2：统计匹配仅用(盘口+差异%+倾向)三个维度，无球队级数据

当前 `get_statistical_prediction()` 的 SQL 查询：
```sql
WHERE cluster = ? AND pan = ? AND trend = ? AND diff BETWEEN ? AND ?
```

完全没有用到：
- 两队近期场均进失球
- 主客场进球画像
- 历史交锋进球规律

**后果**：同样的盘口 2.5 + 倾向"大"，既可能匹配到曼城vs利物浦（实际打出 4 球），也可能匹配到西甲保级队互啄（实际打出 0 球），统计粒度太粗。

### 🔴 缺陷 3：统计模型与 LLM 推理完全割裂，缺乏加权融合

当前流程：
```
统计模型 → 输出 "2, 3球" → 作为"参考"文本传给 LLM
LLM推理 → 输出自己的 "1, 2球" → 覆盖统计结论
```

两者并行输出两个独立的预测值，没有任何数学融合公式。LLM 往往根据自己的基本面判断直接否决统计信号（或反过来盲目跟从统计），缺乏一个量化的加权决策机制。

### 🟡 缺陷 4：进球数专项复盘缺失

`run_post_mortem.py` 和 `2_Post_Mortem.py` 只追踪胜负预测准确率，从未统计进球数预测准确率。预测错了也不知道错在哪，无法形成针对性的复盘→优化闭环。

### 🟡 缺陷 5：`goal_analysis_history` 表无自动回写

每次赛后实际总进球数没有自动写入 `goal_analysis_history`，导致：
- 历史样本增长停滞（目前仅 203 场）
- 统计模型的后验概率无法随赛果更新而进化

### 🟡 缺陷 6：进球数数据依赖人工 Excel 录入

`goals_pan`、`goals_diff_percent`、`goals_trend` 三个关键字段全靠人工填写 `foot_prediction.xlsx`，缺乏爬虫自动化。

## 二、重构方案：四层十维深度融合架构

### 整体架构

```
┌──────────────────────────────────────────────────────────────┐
│  🎯 输出层 (Layer 4): 融合决策引擎                             │
│      D10 统计后验概率 × LLM前向推理 → 加权融合输出             │
│      公式: P(goal=X) = α×P_stat(X) + (1-α)×P_llm(X)         │
├──────────────────────────────────────────────────────────────┤
│  ⚖️ 场景层 (Layer 3): 战意与外部因素                           │
│      D7 比赛性质与战意 | D8 核心伤停 | D9 赛程密度与体能       │
├──────────────────────────────────────────────────────────────┤
│  📊 球队数据层 (Layer 2): 两队攻防画像                          │
│      D4 近期进球趋势 | D5 主客场进失球 | D6 交锋进球规律       │
├──────────────────────────────────────────────────────────────┤
│  🏛️ 机构盘赔层 (Layer 1): 市场定价信号                         │
│      D1 进球盘口 | D2 差异%&倾向 | D3 联赛DNA特征组            │
└──────────────────────────────────────────────────────────────┘
```

### 各维度详细定义

#### D1. 进球数盘口（Goals Pan）
- **数据源**：竞彩官方进球数盘口线（如 2.5、3.0、3/3.5）
- **判读规则**：

| 盘口区间 | 场景 | 判读 |
|:---|:---|:---|
| ≥ 3.5 | 五大联赛豪门参与 | 首要怀疑"诱大"，优先防小球（1-2球） |
| ≥ 3.5 | 荷甲/美职/澳超等大球联赛 | 非诱盘，可能是真实定价 |
| 2.0~2.25 | 强队vs弱队实力悬殊 | "浅盘防穿"，警惕屠杀打出4-5球 |
| 2.5 | 无明显倾向 | 标准盘，以基本面为主 |

#### D2. 预测差异百分比 & 机构倾向
- **数据源**：进球数盘口对应的水位差异和"大/小"倾向标记
- **判读规则**：

| 差异% | 倾向 | 判读 |
|:---|:---|:---|
| > 0.10 | 大 | 机构真实防守大球方向，上调进球预期 |
| > 0.10 | 小 | 机构真实防守小球方向，下调进球预期 |
| ≈ 0 | — | 市场均衡，回归基本面 |

#### D3. 联赛 DNA 特征组（统一到 `constants.py`）
- **5 组定义**（消除当前 3 处不一致）：

| 组别 | 名称 | 包含联赛 | 进攻特征 |
|:---|:---|:---|:---|
| A | 开放大开大合型 | 澳超、挪超、荷甲、荷乙、瑞超、美职足、沙特、芬超 | 场均进球≥2.8，深盘也敢出大球 |
| B | 严密防守型 | 意甲、西乙、法乙、阿甲、葡超、英冠、解放者杯 | 场均进球≤2.2，浅盘也优先防小 |
| C | 主流均衡型 | 英超、德甲、西甲、法甲、德乙 | 1.5~3.5球均可，结合具体赛季风格 |
| D | 日韩孤立型 | 日职、韩职、韩K | 节奏快但战意波动大，独立评估 |
| E | 杯赛/特殊 | 欧冠、欧联、欧协、各国杯赛 | LLM动态判断两队联赛DNA |

#### D4. 近期进球趋势（Form Goals）
- **数据**：主队/客队近6场场均进球
- **已存在数据源**：`recent_form.home` / `recent_form.away` 中的战绩字符串（可正则提取）+ `advanced_stats.avg_goals_for`
- **判读规则**：

| 主队近6场场均 | 客队近6场场均 | 判读 |
|:---|:---|:---|
| > 2.0 | > 1.5 | 双方火力正旺，上调总进球预期 +0.5 |
| > 2.0 | < 1.0 | 主队单方面碾压，总进球取决于主队 |
| < 1.0 | < 1.0 | 双方哑火，强制下调预期 -0.5 |
| — | > 2.0 | 客队火力凶猛，上调总进球 |

#### D5. 主客场进失球画像（Home/Away Profile）
- **数据**：主队主场场均进球+失球 / 客队客场场均进球+失球
- **已存在数据源**：`advanced_stats` 中（如果有 API）或从 `recent_form` 正则提取
- **计算公式**：
  ```
  预期总进球 = (主队主场场均进球 + 客队客场场均失球) / 2
             + (客队客场场均进球 + 主队主场场均失球) / 2
  ```

#### D6. 交锋进球规律（H2H Goals Pattern）
- **数据**：近5次历史交锋的总进球数分布
- **数据源**：`h2h_summary` 字符串（可尝试正则提取比分）
- **判读规则**：

| 近5次交锋特征 | 判读 |
|:---|:---|
| ≥ 4次打出3+球 | 两队碰面互爆，强制上调总进球 |
| ≥ 4次打出0~1球 | 防守克制对局，强制下调总进球 |
| 进失球分布均匀 | 按正常分析 |

#### D7. 比赛性质与战意（Match Context）
- **数据源**：`recent_form.standings`（积分榜位置）+ 联赛阶段判断

| 场景 | 进球倾向 | 判读 |
|:---|:---|:---|
| 保级生死战 | ↓↓ 偏小 | 双方极度保守，优先1~2球 |
| 争冠关键战 | ↑/↓ 极端化 | 要么双方猛攻、要么极端紧张，不取中间值 |
| 德比战 | ↓ 偏小 | 身体对抗激烈，进球偏少但不确定性高 |
| 赛季末无欲无求 | ↑ 偏大 | 放开踢，可能大球 |
| 杯赛淘汰赛 | ↓ 偏小 | 保守为主，加时才放开 |
| 双方均需3分 | ↑ 偏大 | 对攻概率高 |

#### D8. 核心伤停（Injury Impact）
- **数据源**：`recent_form.injuries`
- **量化规则**：

| 伤停类型 | 影响 |
|:---|:---|
| 核心前锋缺阵 | 该队进球预期 -0.5 球 |
| 核心后卫/门将缺阵 | 该队失球预期 +0.5 球 |
| 双方防线核心均缺 | 强制上调总进球预期 |
| 无核心伤停 | 无影响 |

#### D9. 赛程密度与体能（Fatigue Factor）
- **数据源**：`recent_form` 中上一场比赛时间

| 赛程情况 | 判读 |
|:---|:---|
| 一周双赛 / 欧战后仅休2~3天 | 轮换+体能下降，进球偏少 |
| 休息≥5天 | 主力满血，正常发挥 |
| 国家队比赛日后 | 旅途疲劳，进球偏少 |

#### D10. 统计后验概率与 LLM 推理的加权融合
- **核心公式**：
  ```
  最终概率(总进球=X) = α × P_stat(总进球=X) + (1-α) × P_llm(总进球=X)
  ```
- **α 权重表**：

| 历史匹配样本量 | α 权重 | 含义 |
|:---|:---|:---|
| ≥ 15 | 0.50 | 统计可信度高，与 LLM 等权 |
| 10~14 | 0.40 | 统计有一定参考价值 |
| 5~9 | 0.30 | 统计信号偏弱 |
| 2~4 | 0.15 | 统计仅辅助参考 |
| 0~1 | 0.00 | 完全信任 LLM 推理 |

### Phase 2: Prompt 升级（核心，2 天）

#### Step 2.1: 重写 `GoalsPredictor` 的系统 Prompt

**文件**：[src/llm/goals_predictor.py](file:///e:/zhangxuejun/football-prediction/src/llm/goals_predictor.py)

**目标**：将当前松散的 5 步分析改为结构化的四层十维分析流程。

**新 Prompt 结构**：

```
# Role: 进球数专项预测专家

## 核心哲学：机构定价为锚，球队数据为帆
进球数盘口是机构对这场的"定价"，球队进失球数据是实际能打出的"引擎"。
两者必须交叉验证，严禁单维度决策。

## 十维分析框架（必须逐层推演）

### Layer 1: 机构盘赔信号
- 从【机构数据】中提取 D1进球盘口、D2差异%倾向、D3联赛特征组
- 盘口≥3.5+豪门=首要怀疑诱大 → 优先1-2球
- 盘口2.0~2.25+强队vs弱队="浅盘防穿" → 警惕4-5球

### Layer 2: 球队攻防画像
- 从【球队数据】中提取 D4近期场均进失球、D5主客场画像、D6交锋规律
- 双方场均≥2.0→上调，双方场均<1.0→强制下调

### Layer 3: 战意与场景修正
- D7战意 → D8伤停 → D9赛程密度
- 保级生死战强制压小球，核心前锋缺阵扣0.5球

### Layer 4: 统计后验 + LLM前向融合
- 【统计模型提示】中给了历史最高概率进球数
- 如果历史样本≥10场且概率>60%，必须将该进球数作为最终输出的核心之一

## 输出格式（强制）
【核心逻辑推演】（≤150字）
【统计数据洞察】
【进球数预测】 X, Y球（概率：X球 Z%，Y球 W%）| 备选：N球 | 置信度：M%
【进球数推演过程】
  - D1盘口信号: ...
  - D2差异倾向: ...
  - D3联赛特征: ...
  - D4近期趋势: ...
  - D5主客画像: ...
  - D6交锋规律: ...
  - D7战意修正: ↑/↓/—
  - D8伤停修正: ↑/↓/—
  - D9体能修正: ↑/↓/—
  - D10统计融合: α=0.XX, P_stat=(X球:40%, Y球:35%)
  - 最终决策: ...
```

## 六、深度审查：遗漏维度与必须调整点

> 以下是对原始四层十维框架的二次深度审查，识别出的遗漏维度、数据工程缺口和模型评估缺陷。

### 6.1 遗漏的独立预测维度（建议扩展至十七维）

#### D11. 天气条件修正（Weather Impact）

**缺失原因**：当前框架完全未考虑天气对比赛节奏和进球数的影响。

| 天气条件 | 影响机制 | 预期进球修正 |
|:---|:---|:---|
| 大雨/暴雨 | 场地湿滑，皮球滚动受阻，远射和长传精度大幅下降 | **-0.3~0.5 球** |
| 大风（>20km/h） | 影响高空球和射门精度，任意球头球威胁降低 | **-0.2~0.3 球** |
| 极高温（>35°C） | 比赛节奏放缓，补水暂停增多，下半场进球减少 | **-0.3 球** |
| 雪战 | 极度不利于技术型球队，进球极少 | **-0.5+ 球** |
| 晴好/室内 | 正常发挥 | 无修正 |

**数据可行性**：可从比赛城市天气预报 API 获取，也可从 `match_data` 的场地信息推断（室内/室外）。

#### D12. 半场进球率与进球时间分布

**缺失原因**：同样的全场 3 球，"上半场 3-0 后下半场散步"和"上半场 0-0 下半场 3 球对攻"的预测逻辑完全不同。

- **D12a 半场进球占比**：主队/客队上半场进球占全场进球的比率。如果某队上半场进球占比 > 60%，说明进入状态快，全场大球概率更高。
- **D12b 上半场破僵率**：如果两队近期比赛中上半场丢球率均高，全场上半场就可能打破 0-0 僵局，降低 0-0 完场概率。

**数据可行性**：中低。需要半场比分数据，当前爬虫不抓取。可降级为：通过经验规则（如荷甲/德甲上半场进球率高）作为 Prompt 中的联赛特性提示。

#### D13. 让球盘深度与进球盘口的交叉矛盾信号

**缺失原因**：当前 D1 只分析进球盘口本身，D3 只看联赛，但让球盘（亚指）和进球盘口之间的"矛盾"往往是强力信号。

| 让球盘 | 进球盘口 | 信号含义 |
|:---|:---|:---|
| 深盘（-1.5 以上） | 浅进球盘（2.25~2.5） | 机构认为强队会赢但不会屠杀 → **赢球输盘 + 小球** |
| 深盘（-1.5 以上） | 深进球盘（3.0+） | 机构认为强队会大屠杀 → 正路大球 |
| 浅盘（-0.25~0.5） | 深进球盘（3.0+） | 双方实力接近但预期对攻 → 大球>胜负 |
| 浅盘（-0.25~0.5） | 浅进球盘（2.0~2.25） | 菜鸡互啄 → 小球 |

**数据可行性**：高。让球盘（亚指）数据已有，进球盘口数据来自 Excel/爬虫，只需在 Prompt 中增加交叉判读逻辑。

#### D14. 射门转化率（Shot Conversion Rate）

**缺失原因**：只看场均进球不看射门数，无法区分"运气好"和"真实力"。一支场均射门 15 次只进 1 球的球队，比场均射门 5 次进 1 球的球队有更大的大球潜力（射门数迟早回归进球）。

- 转化率 > 18% → 进攻高效，但可能回落
- 转化率 < 8% → 进攻低效，但可能反弹回归均值
- 对方被射门转化率高（>15%）→ 防线质量差，对方容易进球

**数据可行性**：低。需要射门数据，依赖 `advanced_stats` API 且该 API 可能不提供。可降级为 Prompt 中的"注意：如果某队近期进球数明显高于/低于其创造的机会数，预期有回归均值的可能"。

#### D15. 零封概率与主客队进球拆解

**缺失原因**：当前框架只预测"总进球数"，但实际进球数的构成（2-0 vs 1-1）在分析中更有信息量。

- 如果强队主场让深盘且进球盘口 2.5→ 更可能打出 **2-0 / 3-0**（强队零封），而非 **1-1 / 2-2**（双方进球）
- 如果两队近期防守都差且盘口不深 → 更可能 **1-1 / 2-2 / 2-1**（双方都有进球）
- **建议**：在 Prompt 中增加一个子任务——先判断"是否双方都进球（BTTS）"，再推导总进球数，比直接猜总进球更精确。

**数据可行性**：高。主客队场均失球数据可从 `recent_form` 正则提取或 `advanced_stats` 获取。

#### D16. 联赛赛季阶段（Season Stage）

**缺失原因**：同一联赛在赛季不同阶段，进球率有系统性差异。

| 阶段 | 特征 | 进球倾向 |
|:---|:---|:---|
| 前 8 轮 | 球队磨合不足，防守漏洞多，新援冲劲强 | ↑ 偏大 |
| 中段（9~28 轮） | 稳定期，按正常分析 | 正常 |
| 最后 5 轮 | 保级队死守、争冠队紧张，无欲无求队放开踢 | 极端化（小球与大球并存） |
| 最后 2 轮 | 提前降级/夺冠球队"垃圾时间" | 不确定性最高 |

**数据可行性**：高。从比赛日期和联赛开赛日期即可推算，可在 `_format_match_data` 中通过日期计算自动标注。

#### D17. 球队风格匹配度（Style Compatibility）

**缺失原因**：当前按联赛分组（Cluster A~E）太粗。同属 A 组的两支大球联赛球队相遇，如果都是控球型，反而可能慢节奏。

| 主队风格 | 客队风格 | 预期节奏 | 总进球倾向 |
|:---|:---|:---|:---|
| 控球 | 控球 | 慢（双方争夺控球权） | ↓ 偏小 |
| 控球 | 反击 | 一攻一守，空间大 | ↑ 偏大 |
| 反击 | 反击 | 双方互偷，回合多 | ↑ 偏大 |
| 高位压迫 | 高位压迫 | 失误多，转换快 | ↑ 偏大 |
| 防守反击 | 防守反击 | 双方收缩 | ↓↓ 偏小 |

**数据可行性**：低。球队风格标签需要额外数据源。可降级为 Prompt 中的定性提示："注意分析两队的比赛风格是否互克"。

### 6.2 盘赔维度的深层信号遗漏

#### 遗漏 1：进球数盘口的水位变化斜率

当前 D2 只用了"预测差异百分比"这一个静态快照值，没有捕获水位从初盘到即时盘的**动态变化趋势**：

| 水位变化 | 信号含义 |
|:---|:---|
| 大球水位 0.95→0.78 (急降) | 机构真怕大球打出，在阻挡买入 → **上调进球预期** |
| 大球水位 0.85→1.02 (急升) | 机构在诱导大球，不怕赔付 → **下调进球预期** |
| 水位无波动 | 市场均衡，回归基本面 |

**建议**：在 `_format_match_data` 中，如果能从爬虫拿到初盘→即时盘的大球水位变化，需作为 D2 的补充子维度传入 Prompt。如果爬虫拿不到（只有 Excel 一个静态差异%），在 Prompt 中标注"水位变化数据缺失"。

#### 遗漏 2：角球盘口作为辅助校验信号

角球多 = 进攻回合多 = 射门多。角球盘口异常高（如 ≥ 10.5）通常预示着高节奏对攻战。虽然角球不是直接预测进球数的数据，但可以作为 Prompt 中的一个辅助校验：当 LLM 输出"小球"但角球盘口极高时，触发自我矛盾检查。

**数据可行性**：如果能从 500.com 抓取角球盘口（playid 未知），可作为辅助维度。否则放弃。

### 6.3 数据工程可行性缺口

#### 缺口 1：Phase 1.2（进球数盘口爬虫）优先级严重偏低

当前文档将进球数盘口自动抓取标记为"如可行，维持现状不算致命"，**但这是整个系统的数据入口命脉**。没有自动化盘口数据，整个模型永远依赖人工 Excel 录入，无法规模化。

**具体方案**：
- 500.com 的竞彩进球数盘口对应的 `playid` 有待调研确认（通常为 270 或 271）
- 如果 trade.500.com 可抓，在 `OddsCrawler` 或 `JingcaiCrawler` 中新增 `fetch_goals_pan(fixture_id)` 方法
- 如果 Web 端反爬，可考虑用 `autoglm-browser-agent` 技能模拟浏览器抓取

**建议**：将 Phase 1.2 从"如可行"提升为 **P0 必须完成**。

#### 缺口 2：D6（交锋进球规律）数据源不可行

文档计划从 `h2h_summary` 字符串正则提取近 5 次交锋的比分。但实际爬虫返回的 `h2h_summary` 格式为纯文本描述（如"近5次交锋主队2胜1平2负"），**不包含具体的比分数字**。D6 的"近5次交锋总进球数分布"在当前数据结构下**完全无法计算**。

**替代方案**：
1. 如果能从 500.com 的历史交锋页面抓取具体比分（需要新的爬虫端点），可以实现
2. 短期内降级为：只用 h2h_summary 中的"胜负分布"作为 D6 的定性输入（如"交锋双方互有胜负，进球偏多"），不做精确统计

**建议**：将 D6 标记为"依赖新爬虫端点，Phase 2 中降级为定性信号"。

- 无 h2h → 跳过 D6

### 6.4 模型评估与迭代闭环缺失

#### 缺口 1：进球数预测评估指标不完整

文档只提到"精确匹配 + ±1球命中率"，但进球数预测的评估需要更精细的指标体系：

| 指标 | 计算方式 | 用途 |
|:---|:---|:---|
| 精确命中率 | 预测值 = 实际值 | 核心 KPI |
| ±1 球命中率 | \|预测-实际\| ≤ 1 | 容错 KPI |
| 平均绝对误差 (MAE) | mean(\|预测中位数-实际\|) | 模型校准度 |
| 双选覆盖率 | 双选包含实际值的比例 | 双选质量 |
| 按盘口区间统计 | 2.0 / 2.5 / 3.0 / 3.5 分别计算 | 定位哪类盘口是盲区 |
| 按特征组统计 | A/B/C/D/E 组分别计算 | 定位哪类联赛是盲区 |

#### 缺口 2：缺少 A/B 测试框架

改造后的新 Prompt vs 旧 Prompt，如何验证哪个版本更准？

**建议**：在 Dashboard 的进球数控制面板中，增加一个"对比预测"按钮，对同一批比赛同时跑旧 Prompt 和新 Prompt，对比两者的精确命中率。

#### 缺口 3：统计模型自学习闭环缺少触发设计

Phase 1.3 写了自动回写，Phase 3.3 写了自动刷新 `goal_distribution_analysis.md`。但缺少触发时机设计：
- 每次 `run_post_mortem.py` 执行后自动触发 `temp_analyze.analyze()`
- 或者 Dashboard 的复盘页面生成复盘报告时自动触发

### 6.5 D10 加权融合公式的精确化调整

文档中的公式 `P_final = α × P_stat + (1-α) × P_llm` 有两个需要精确化的地方：

#### 调整 1：α 应按（联赛组, 盘口区间）二维查询，而非仅按样本量

同样是 10 场历史样本，A 组（澳超，进球波动大）的统计可信度低于 B 组（防守联赛，进球稳定）。建议：

```
α = 基础样本量权重 × 联赛组稳定系数
```

| 联赛组 | 稳定系数 | 说明 |
|:---|:---|:---|
| A 组（大开大合） | 0.70 | 进球波动大，统计均值代表性弱 |
| B 组（严密防守） | 1.10 | 进球稳定，统计可信度高 |
| C 组（均衡） | 1.00 | 基准 |
| D 组（日韩） | 0.85 | 战意干扰大 |
| E 组（杯赛） | 0.60 | 样本极少 |

#### 调整 2：LLM 概率分布的数值化提取

LLM 输出的"进球数预测: 2, 3球（概率：2球 60%，3球 30%）"是自然语言，不是数值。需要：
1. 在 Prompt 输出格式中，强制要求 LLM 以固定格式输出概率（如 `|2球:0.60|3球:0.30|`），便于正则提取
2. 或在 `predict()` 返回的 JSON 中要求 LLM 返回 `{"probabilities": {"2": 0.60, "3": 0.30}}`
